# 代码与安全审计报告（2026-07-19）

## 范围与原则

审计覆盖 FastAPI 后端、Vue 管理端、静态数据大屏、Docker Compose、Nginx 和直接/间接依赖。清理遵循以下边界：

- 不删除、改名或新增现有业务 API 路由。
- 不改变腾讯表格字段映射、统计口径、轮询时间窗口和大屏交互。
- 只直接实施可由回归测试证明兼容的修改。
- 会改变现网登录、公开大屏或 Token 设置方式的安全项仅记录，不擅自启用。

## 本次已处理

### 正确性

- 修复 Axios 并发 401 刷新死锁。原实现会让发起刷新请求的第一个业务请求在刷新成功后才加入等待队列，从而永久挂起；现改为所有请求共享同一个刷新 Promise。
- 损坏的 `localStorage.user` 不再导致管理端初始化崩溃，会自动清除并回到可恢复状态。
- 腾讯轮询时间改用带时区的 UTC 时间，同时兼容数据库内已有的无时区历史时间戳。
- 前端轮询状态类型与后端实际响应字段对齐。

### 死代码与依赖

- 删除已下线 Univer 本地编辑器遗留的保存/加载 Schema、角色依赖工厂和不可达大屏渲染分支。
- 删除不可达分支对应的无用 CSS、函数和无用导入。
- 删除运行时未引用的 `python-multipart`、Alembic 和 Redis 服务。
- `openpyxl` 移到 `requirements-migration.txt`，只供一次性 Excel 迁移脚本使用。
- Docker 前端构建改用 `npm ci`，保证按锁文件复现。
- 重写已经与当前架构不一致的 README。

### 安全

- 腾讯 `access_token` 使用 Fernet 加密落库；旧明文配置仍可读取，并在下次保存时自动迁移。
- 通配 CORS 不再同时允许跨站凭据。
- Nginx 增加 CSP、`nosniff`、同源嵌套、Referrer Policy 和浏览器权限限制。
- Vite 开发服务器默认只监听本机，降低开发依赖漏洞的暴露面。
- 启动时对默认 JWT 密钥、默认管理员密码和通配 CORS 输出明确警告。
- FastAPI/Pydantic/PyJWT/cryptography 升级到通过当前漏洞库扫描的版本。

## 未直接改变的风险

### 严重：腾讯控制接口未鉴权

以下写接口当前是公开的：

- `PUT /api/tencent/config`
- `POST /api/tencent/sync`
- `POST /api/tencent/poll/set`
- `POST /api/tencent/poll/trigger`

攻击者可替换同步配置、触发腾讯 API 调用或消耗调用额度。现有大屏 Token 输入框依赖这一公开行为，直接加 JWT 会破坏当前使用方式，因此本次未擅自修改。

建议下一阶段把设置入口接入管理员登录，或增加独立管理密钥并由 Nginx 对来源 IP 限制。Webhook 路由应继续保持公开，但必须保留现有腾讯 RSA 签名校验。

### 严重：公开注册账户权限偏大

`POST /api/register` 公开可用，注册用户默认获得 `employee` 角色；该角色包含表格更新能力。文档列表与部分操作主要按 RBAC 权限判断，而不是始终按 owner/project 隔离。

建议在确认账号流程后选择其一：关闭公网注册、改为管理员邀请、注册后默认禁用等待审核，或把默认角色改成只读。此项会改变现网账号流程，需单独实施和验收。

### 高：生产凭据仍有兼容默认值

Compose 已支持通过环境变量覆盖数据库密码和管理员密码，但为避免当前服务器无配置时中断，仍保留了开发默认值。上线前必须显式设置：

- `POSTGRES_PASSWORD`
- `JWT_SECRET`
- `SUPERADMIN_PASSWORD`

本次沟通中曾直接提供过服务器密码和腾讯 Token，这些值应视为已暴露并轮换。切换 `JWT_SECRET` 后需要重新录入腾讯 Token，因为它同时派生落库加密密钥。

### 高：缺少接口限流和操作审计

登录、注册、手动同步和轮询触发接口均无频率限制；配置修改也没有记录操作者、来源 IP 和变更内容。建议在 Nginx 对认证/同步接口增加限流，并新增不可变的管理操作审计表。

### 中：浏览器 Token 存于 localStorage

现状容易受前端 XSS 影响。CSP 已降低风险，但彻底解决需要改为 `HttpOnly + Secure + SameSite` Cookie，并增加 CSRF 防护。这会改变认证接口与部署方式，应独立规划。

### 中：单进程假设明确

轮询锁、API 调用计数和大屏缓存部分依赖单个 Uvicorn worker。当前 Docker 明确使用一个 worker，因此行为正确；以后横向扩容前，需要把锁、任务调度和计数迁移到 PostgreSQL 锁或独立任务队列。

### 中：开发工具链漏洞

生产依赖的 `npm audit --omit=dev` 为零漏洞；完整审计仍会报告 Vite 5 开发服务器问题，上游自动修复要求跨多个主版本升级。当前通过仅监听 `127.0.0.1` 降低暴露面，建议在单独前端升级任务中迁移到 Vite 8 并做浏览器回归。

## 后续结构优化建议

1. 将 `routers/tencent.py` 拆为配置、同步、轮询、Webhook 四个边界，路由只负责校验和响应，业务统一进入同步协调器。
2. 为统计接口补全 `response_model`，并从同一套 Pydantic 模型生成前端类型，减少前后端字段漂移。
3. 将目前的独立脚本测试迁移到 pytest fixture，隔离数据库并加入 API 契约快照。
4. 对大数据清单增加服务端分页/游标，避免未来把数千条明细一次性送到浏览器。
5. 增加结构化日志、同步批次 ID、耗时、读取范围、写入/删除数量和腾讯请求 ID，便于定位缓存与全量同步问题。
6. 当前建表依赖 SQLAlchemy `create_all` 与 seed；下一次正式改表前应恢复真正使用的迁移工具，而不是只把 Alembic 作为未配置依赖安装在镜像中。
